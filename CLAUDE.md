## Cache-busting discipline

Every local `js/`/`css/`/`img` asset is loaded with a `?v=<hash>` query in the HTML, where `<hash>` is the first 8 hex of the asset's content sha1. **`npm run stamp` (`tools/stamp-assets.mjs`) rewrites these from the file bytes and is part of `npm run build`** — editing an asset and rebuilding updates every page that loads it automatically; there is no manual `?v=N` to bump. `npm run check:fresh` (`tools/check-fresh.mjs`) re-runs the whole pipeline and fails if the committed `js/sheet.js`, `js/trainer.js`, or any HTML stamp is stale — wire it into CI / run it before committing. (Note: `?v=` queries embedded in CSS `url()`s, e.g. `img/logo.svg` in `css/site.css`, are not stamped — only HTML refs are.)

## Architecture

### Data flow (single authoring source)

```
data/fto_algs.json   ← authored algorithm list (the live TCP + 1L3T + LBT + 1LP authority)
   │ npm run build:sheet                          ↖ will be fetched at runtime by the pages at M3+
   ▼
js/sheet.js  (SHEET = { ALG / NAME / CNAME / PRES })  +  data/classmap.json   ← generated data-quality gates; no page consumes them at runtime
```

The live runtime consumers (`js/algs.js` since M3, the trainer bundle since M4, `js/solver-core.js`'s finish index since M5) fetch `data/fto_algs.json` at runtime. The compiled `js/sheet.js` + `data/classmap.json` remain the sheet pipeline's machine-checked data-quality gate (`npm run check`). Alg texts display VERBATIM (site convention); `engine.normAlg` is the canonical spelling where one is needed. FTO sheet keying as of M1: renderKey = `stateKey` (the full state), canon = `realCanonKey` = **identity fold** — the case-symmetry fold (subgroups of the 24 rotations, via conjugation + recoloring) is decided WITH the M3 sheet data; widening it later must re-key the compiled sheet.

### The engine is the geometric core (FTO as of M1)

`js/engine.js` (`window.OOEngine`) is the **FTO engine**. State `{ cp:[6], co:[6], ep:[12], ctr:[24] }`: corner perm + 0/1 flips over vertex slots [+x,+y,+z,−x,−y,−z], edge perm (no orientation), centre COLORS per slot (identical triplets, two never-mixing 12-slot orbits). Everything derives from geometry at init: faces = sign vectors on the unit octahedron, cut planes at 1/3 depth, clockwise = right-hand −θ about the outward axis; the facelet scheme (72, `9*face + pos`, faces [U,F,BR,BL,L,R,D,B]) is IDENTICAL to xyzzy's ftosolver.js by construction and pinned against it in tests. Layers: 16 native moves (8 faces × 2 directions) as facelet perms + derived slot tables (`moveTables`); Streeter-notation parser (`w`/`s`/`o`/`T` tokens; slices spelled U/R/F/L only); `applyParsed` resolves letters through a FRAME (whole-puzzle rotations never touch the state — a pure rotation has identity effect); the invertible slot-table layer (`effectTable`/`invertTable`/`applyTable`) backs `caseStateOf`/`algSolvesKey` for rotation-containing algs (centres invert as SLOT perms, sidestepping identical-piece ambiguity). Wide/slice desugar: `Xw = Xo ∘ OPP(X)`, `Xs = Xo ∘ X′ ∘ OPP(X)` (opposite face UNPRIMED — machine-verified). Dropped from the Skewb contract: `idx`/`unidx`/`NSLOTS` (no dense full-space indexing at 3.1e22), `optimalSolution`/`optimalScramble` (per-step search returns at M5), `buildSyms`/`makeCanon` family, `enumFreeSlots`, `prependAUF`, `inverseState`, the WCA/NS converters. Added: the facelet/table layers above, `ROT24`, `sliceFaceletPerm`, `rotFaceletPerm`, `randomScramble(len, rng)` (random-move, suppression rules; random-state stays deferred — see the M5 plan entry), `stateSpaceCount()` (BigInt). `render.js`, `account.js`, `navbar.js`, `config.js` are unchanged-inherited; `tables.js` is the M5 pattern-database layer (the Skewb full-state BFS members were deleted with it).

### The solver (M5: Bencisco per-step search + sheet finishes)

`js/tables.js` (`window.OOTables`) owns the coordinate codecs and the Int8 distance-to-goal PDBs (families B/A/C/E3/E6/H1; IndexedDB-cached under 'fto-tables'/'fto-pdb-v2' — bump the key when the payload shape changes; Node tools call `buildPDBs(E)` directly). `js/solver-core.js` (`window.OOSolverCore.makeSolverCore(E, T, PDB, algData)`) is the pipeline: per-step weighted IDA* (stages fc/t1/t2/sc/c3/c4) → LBT effect-match → L3T exact-match, orientation ladder via conjugation, and `verifyLine` (the end-to-end applyParsed proof every emitted item passes; `search()` silently drops and counts failures — `verifyFailures` must stay 0). Core init self-asserts the geometry, the rotation spellings, and the conjugation/frame-walk agreement — a bad change throws at load, not per solve. `js/solver.js` is the page app (boot builds/loads PDBs with progress, search runs synchronously with a UI pause note). Gotchas: LBT case states are only meaningful through their EFFECT tables (exact-key matching misses by design — see ground-truth §Methods); step caps interact with the IDA* weight (a length-L solution may need bound ≈ w·L, handled inside `searchStep`); the search is deterministic (no Math.random anywhere in the pipeline), so lab/test results reproduce.

### The sheet compiler (hermetic; FTO re-key pending M3)

`tools/compile-sheet.mjs` compiles the JSON through engine helpers only and writes two outputs: the `SHEET` data line of `js/sheet.js` (the file is read as a write-template to preserve its surrounding code) and `data/classmap.json` (canonical case key → lowercased subset id, generated from subset membership — never hand-maintain it). The carry-forward mechanism survives from upstream but starts empty: baseline = **committed snapshot `data/prior-sheet.json`** (currently `{}` — the JSON is the sole authority; re-baseline by overwriting it with a freshly built sheet if the JSON ever stops reproducing shipped cases you want kept), and the broken-alg allowlist `data/broken-algs.json` (currently `[]`; an array of `{renderKey, algorithm, reason}`, read by both the compiler and `check-sheet.mjs`). The compiler self-checks every emitted alg and refuses to write outputs that fail; a non-solving alg that isn't in the manifest fails both tools (`UNEXPECTED BROKEN`/`BROKEN`), a *new* unparseable alg fails the build, and an alg authored under a different case than it geometrically solves is an `ADVISORY`. `tools/check-sheet.mjs` re-validates the shipped sheet but shares the engine's keying functions, so it is not independent of engine-level keying bugs.

### Module strategy (why no `"type": "module"`)

The `js/*.js` files are **classic browser scripts** that attach to `window` (`OOEngine`, `OOSheet`, `OORender`, `OOAccount`). The Node tools (`.mjs`, ESM) load them for their **side effect**, not their return value: they set `globalThis.window = {}`, `require()` the file, then read `globalThis.window.OOEngine`. Each `js/` file wraps its body in an IIFE with a local `const module = {exports:{}}` that shadows Node's real `module`. Adding `"type": "module"` to package.json would break the tools' `require()` calls — it is intentionally omitted.
