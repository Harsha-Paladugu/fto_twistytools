fto.twistytools.com (like its parents Skewbiks.com and Pyraminx.net) is a **static, no-build-required website**. Four HTML pages share one engine and a set of UI layers. ## Commands

```bash
npm install
npm run build         # full pipeline: build:trainer (sheet + trainer bundle), then stamp asset hashes, then npm run check
npm run build:trainer # build:sheet, then esbuild-bundle the trainer (node build.mjs) → js/trainer.js
npm run build:sheet   # compile data/fto_algs.json → js/sheet.js + data/classmap.json (empty seed until M3)
npm run watch:trainer # esbuild watch on src/trainer (node build.mjs --watch) — does NOT recompile the sheet
npm run stamp         # rewrite each asset's ?v= query to its content hash (tools/stamp-assets.mjs)
npm run check         # verify the shipped js/sheet.js against the engine (alias: npm test)
npm run check:fresh   # re-run the pipeline; fail if js/sheet.js, js/trainer.js or any HTML stamp is stale
npm run test:engine   # FTO engine tests (tools/test-engine.mjs) — 37 tests, oracle-pinned
npm run test:trainer  # KNOWN-RED until M4 (Skewb substrate; imports the deleted BFS lib)
npm run test:solver   # KNOWN-RED until M5 (same reason)

node tools/compile-sheet.mjs --check   # dry-run compile + report; never writes js/sheet.js
```

The esbuild step lives in `build.mjs` at the repo root (entry `src/trainer/index.jsx` → `js/trainer.js`); there is no separate `build:trainer`-only esbuild script — `node build.mjs` is it.

- There is **no unit-test framework**, but dependency-light test runners exist. `npm test` is aliased to `tools/check-sheet.mjs` (validates the compiled data sheet only). `npm run test:engine` (`tools/test-engine.mjs`) asserts the FTO engine — every face-move facelet table pinned exactly against xyzzy's ftosolver.js fixtures (`tools/fixtures/xyzzy-fto.mjs`), rotation/mirror pins, slice/wide desugar identities, Streeter-notation parse/normalize/invert/mirror, state↔facelet round trips, frame machinery, keying edge cases, parity/orbit invariants, corner sub-space BFS (11,520) and centre-orbit BFS (369,600), the published state-space figure, and scrambles. `tools/test-trainer.mjs` / `tools/test-solver.mjs` are the SKEWB suites and stay known-red until M4/M5 rewrite them. To exercise the engine ad-hoc in Node, replicate what the tools do: `globalThis.window = {}; require('./js/engine.js'); const E = globalThis.window.OOEngine;` (see the "module strategy" gotcha below).
- **To preview the site locally you must serve over HTTP** (e.g. `npx serve` or `python -m http.server`), not `file://` — `js/account.js` uses dynamic `import()` and the Algorithms page `fetch`es the JSON, both of which need an origin.

## Generated files — do not hand-edit

- **`js/sheet.js`** and **`data/classmap.json`** are generated by `tools/compile-sheet.mjs` from `data/fto_algs.json`.
- **`js/trainer.js`** is the minified esbuild bundle of `src/trainer/` (the source of truth is `src/trainer/skewb-trainer.jsx` + `skewb-core.mjs`; `js/trainer.js` is build output, even though it's the file an editor may surface).

Both are committed so the site works on the host without a build. Edit the *sources*, then rebuild.

## Cache-busting discipline

Every local `js/`/`css/`/`img` asset is loaded with a `?v=<hash>` query in the HTML, where `<hash>` is the first 8 hex of the asset's content sha1. **`npm run stamp` (`tools/stamp-assets.mjs`) rewrites these from the file bytes and is part of `npm run build`** — editing an asset and rebuilding updates every page that loads it automatically; there is no manual `?v=N` to bump. `npm run check:fresh` (`tools/check-fresh.mjs`) re-runs the whole pipeline and fails if the committed `js/sheet.js`, `js/trainer.js`, or any HTML stamp is stale — wire it into CI / run it before committing. (Note: `?v=` queries embedded in CSS `url()`s, e.g. `img/logo.svg` in `css/site.css`, are not stamped — only HTML refs are.)

## Architecture

### Data flow (single authoring source)

```
data/fto_algs.json   ← authored algorithm list (the authority; EMPTY SEED until the M3 sheet import)
   │ npm run build:sheet                          ↖ will be fetched at runtime by the pages at M3+
   ▼
js/sheet.js  (SHEET = { ALG / NAME / CNAME / PRES })  +  data/classmap.json   ← generated data-quality gates; no page consumes them at runtime
```

The runtime consumers (`js/algs.js`, the trainer bundle, `js/solver-core.js`) still fetch the DELETED `data/skewb_algs.json` — they are inherited Skewb code, offline until M3/M4/M5 repoint and rewrite them. The compiled `js/sheet.js` + `data/classmap.json` remain the sheet pipeline's machine-checked data-quality gate (`npm run check`). Display notation will be normalized everywhere by the shared `engine.normAlg`. FTO sheet keying as of M1: renderKey = `stateKey` (the full state), canon = `realCanonKey` = **identity fold** — the case-symmetry fold (subgroups of the 24 rotations, via conjugation + recoloring) is decided WITH the M3 sheet data; widening it later must re-key the compiled sheet.

### The engine is the geometric core (FTO as of M1)

`js/engine.js` (`window.OOEngine`) is the **FTO engine**. State `{ cp:[6], co:[6], ep:[12], ctr:[24] }`: corner perm + 0/1 flips over vertex slots [+x,+y,+z,−x,−y,−z], edge perm (no orientation), centre COLORS per slot (identical triplets, two never-mixing 12-slot orbits). Everything derives from geometry at init: faces = sign vectors on the unit octahedron, cut planes at 1/3 depth, clockwise = right-hand −θ about the outward axis; the facelet scheme (72, `9*face + pos`, faces [U,F,BR,BL,L,R,D,B]) is IDENTICAL to xyzzy's ftosolver.js by construction and pinned against it in tests. Layers: 16 native moves (8 faces × 2 directions) as facelet perms + derived slot tables (`moveTables`); Streeter-notation parser (`w`/`s`/`o`/`T` tokens; slices spelled U/R/F/L only); `applyParsed` resolves letters through a FRAME (whole-puzzle rotations never touch the state — a pure rotation has identity effect); the invertible slot-table layer (`effectTable`/`invertTable`/`applyTable`) backs `caseStateOf`/`algSolvesKey` for rotation-containing algs (centres invert as SLOT perms, sidestepping identical-piece ambiguity). Wide/slice desugar: `Xw = Xo ∘ OPP(X)`, `Xs = Xo ∘ X′ ∘ OPP(X)` (opposite face UNPRIMED — machine-verified). Dropped from the Skewb contract: `idx`/`unidx`/`NSLOTS` (no dense full-space indexing at 3.1e22), `optimalSolution`/`optimalScramble` (per-step search returns at M5), `buildSyms`/`makeCanon` family, `enumFreeSlots`, `prependAUF`, `inverseState`, the WCA/NS converters. Added: the facelet/table layers above, `ROT24`, `sliceFaceletPerm`, `rotFaceletPerm`, `randomScramble(len, rng)` (random-move, suppression rules; random-state arrives with M5), `stateSpaceCount()` (BigInt). The other `window.*` layers (`render.js`, `tables.js`, `account.js`, `navbar.js`, `config.js`) are unchanged-inherited; `tables.js`'s full-state BFS members are dead code deleted at M5.

### The sheet compiler (hermetic; FTO re-key pending M3)

`tools/compile-sheet.mjs` compiles the JSON through engine helpers only and writes two outputs: the `SHEET` data line of `js/sheet.js` (the file is read as a write-template to preserve its surrounding code) and `data/classmap.json` (canonical case key → lowercased subset id, generated from subset membership — never hand-maintain it). The carry-forward mechanism survives from upstream but starts empty: baseline = **committed snapshot `data/prior-sheet.json`** (currently `{}` — the JSON is the sole authority; re-baseline by overwriting it with a freshly built sheet if the JSON ever stops reproducing shipped cases you want kept), and the broken-alg allowlist `data/broken-algs.json` (currently `[]`; an array of `{renderKey, algorithm, reason}`, read by both the compiler and `check-sheet.mjs`). The compiler self-checks every emitted alg and refuses to write outputs that fail; a non-solving alg that isn't in the manifest fails both tools (`UNEXPECTED BROKEN`/`BROKEN`), a *new* unparseable alg fails the build, and an alg authored under a different case than it geometrically solves is an `ADVISORY`. `tools/check-sheet.mjs` re-validates the shipped sheet but shares the engine's keying functions, so it is not independent of engine-level keying bugs.

### Module strategy (why no `"type": "module"`)

The `js/*.js` files are **classic browser scripts** that attach to `window` (`OOEngine`, `OOSheet`, `OORender`, `OOAccount`). The Node tools (`.mjs`, ESM) load them for their **side effect**, not their return value: they set `globalThis.window = {}`, `require()` the file, then read `globalThis.window.OOEngine`. Each `js/` file wraps its body in an IIFE with a local `const module = {exports:{}}` that shadows Node's real `module`. Adding `"type": "module"` to package.json would break the tools' `require()` calls — it is intentionally omitted.

### Auth (demo mode; Firebase deferred to M6)

`account.js` exposes one shared sign-in session site-wide, with a `localStorage` "demo mode" fallback when `config.js` has no Firebase config — which is the CURRENT state (`firebase: null`; no Firebase project exists for this site). `firestore.rules` is dormant: just the per-user `users/{uid}` rule, kept as the reviewed baseline for the optional M6 accounts milestone (see `SETUP.md`). The census-era collections/rules/tests live in git history before the M0 commit. Admin edits on the Algorithms page (when M3 revives it) are a per-browser localStorage draft until **Export JSON** → commit → rebuild; there is no live shared store.
